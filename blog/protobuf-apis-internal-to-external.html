Are Protobuf APIs Just for Internal Use? The Connect Inflection Point

Most protobuf APIs are internal today. But Connect protocol is changing that—and context.json is the missing piece for AI-native external APIs.

If you're building with Protocol Buffers, you've probably noticed something: almost everyone uses them for internal services. The moment you need to expose an API externally—to partners, third-party developers, or the public—the advice is usually "just use REST."

But that's changing. And if you're thinking about AI agents consuming your APIs, the change matters more than you might think.

The Current State: 90% Internal

Here's the honest breakdown of protobuf usage today:

Usage Pattern Prevalence Examples
Internal microservices ~90% Service-to-service within org
Mobile → Backend Common App talking to its own servers
Public cloud APIs Growing Google Cloud, some AWS services
External partner APIs Rare Fintech, gaming, IoT

This isn't because protobuf is bad for external APIs. It's because of friction:

  • Tooling complexity — External developers need codegen, proto files, and language-specific setup
  • Browser limitations — gRPC doesn't work natively in browsers (grpc-web is a workaround, not a solution)
  • REST inertia — External developers expect curl-friendly JSON endpoints
  • Documentation — REST/OpenAPI has better tooling for external developer portals

So companies build with protobuf internally for the performance and type safety, then expose a separate REST API externally. Two interfaces, two sets of docs, two maintenance burdens.

Connect: The Inflection Point

The Connect protocol from Buf changes this equation.

Connect is simple: gRPC services that also speak HTTP/JSON natively. Same proto definition, but clients can choose their transport:

  • Internal services → Use binary gRPC for efficiency
  • Browser clients → Use HTTP/JSON, no grpc-web needed
  • External developers → Use curl, standard HTTP clients
  • AI agents → Use whatever works

One schema. Multiple transports. No translation layer.

What this means: Companies can now build with protobuf from day one and expose the same APIs externally without maintaining a separate REST interface. The "internal vs. external" distinction starts to dissolve.

The Missing Piece: Semantic Context

Here's where it gets interesting for AI.

REST APIs have a well-established pattern for machine readability: OpenAPI specs with human-written descriptions. When an AI agent needs to understand a REST API, it reads the descriptions, examples, and documentation.

Protobuf has... comments in proto files.

// User represents a registered user in the system.
// The email field must be unique across all users.
message User {
  string id = 1;
  string email = 2;      // Must be valid email format
  string name = 3;
  int64 created_at = 4;  // Unix timestamp, milliseconds
}

These comments help human developers, but they're unstructured. An AI agent reading this schema can see the structure—fields, types, relationships—but can't reliably understand the semantics:

  • What does created_at represent? Unix timestamp in seconds or milliseconds?
  • Is email case-sensitive?
  • Can name contain Unicode?
  • What happens if you try to create a user with a duplicate email?

The AI has to guess. Or read external documentation. Or make assumptions that might be wrong.

context.json: The Semantic Layer

This is what context.json solves. It's a machine-readable semantic layer that sits alongside your proto files:

{
  "@context": "https://protobuf.ai/ontology/v1",
  "schema": "user.proto",
  "messages": {
    "User": {
      "semanticType": "schema:Person",
      "fields": {
        "email": {
          "maps_to": "schema:email",
          "format": "RFC 5322",
          "unique": true,
          "case_sensitive": false
        },
        "created_at": {
          "maps_to": "schema:dateCreated",
          "unit": "milliseconds",
          "format": "unix_timestamp"
        }
      }
    }
  },
  "edge_cases": [
    "Duplicate email returns ALREADY_EXISTS error",
    "Name supports full Unicode (emoji allowed)"
  ]
}

Now an AI agent can understand your API correctly on the first try. No guessing. No reverse-engineering documentation.

Why This Matters Now

The convergence is happening:

  1. Connect makes protobuf viable for external APIs — The browser/HTTP barrier is gone
  2. AI agents are becoming API consumers — They need machine-readable semantics, not just structure
  3. Schema-first development is winning — Teams want one source of truth, not OpenAPI + proto + docs

If you're building APIs that will be consumed by AI agents—and increasingly, all APIs will be—you need both:

  • Structure (proto files) — What fields exist, what types they have
  • Semantics (context.json) — What those fields mean, how to use them correctly

The Opportunity

For teams already using protobuf internally:

  1. Add Connect — Expose your existing gRPC services over HTTP/JSON
  2. Add context.json — Make your schemas AI-native
  3. Publish to a registry — Let AI agents discover your APIs semantically

Your internal services become external APIs. Your schemas become AI-readable. Your documentation stays in sync because it's generated from the same source.

The "internal vs. external" distinction was always artificial. Connect removes the transport barrier. context.json removes the semantic barrier.

What's left is just APIs—usable by humans, services, and AI agents alike.
