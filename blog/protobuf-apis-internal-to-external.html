Are Protobuf APIs Just for Internal Use? The Connect Inflection Point

Most protobuf APIs are internal today. But Connect protocol is changing that—making protobuf viable for external APIs and AI-assisted development.

If you're building with Protocol Buffers, you've probably noticed something: almost everyone uses them for internal services. The moment you need to expose an API externally—to partners, third-party developers, or the public—the advice is usually "just use REST."

But that's changing. And if you're thinking about AI agents consuming your APIs, the change matters more than you might think.

The Current State: 90% Internal

Here's the honest breakdown of protobuf usage today:

Usage Pattern Prevalence Examples
Internal microservices ~90% Service-to-service within org
Mobile → Backend Common App talking to its own servers
Public cloud APIs Growing Google Cloud, some AWS services
External partner APIs Rare Fintech, gaming, IoT

This isn't because protobuf is bad for external APIs. It's because of friction:

  • Tooling complexity — External developers need codegen, proto files, and language-specific setup
  • Browser limitations — gRPC doesn't work natively in browsers (grpc-web is a workaround, not a solution)
  • REST inertia — External developers expect curl-friendly JSON endpoints
  • Documentation — REST/OpenAPI has better tooling for external developer portals

So companies build with protobuf internally for the performance and type safety, then expose a separate REST API externally. Two interfaces, two sets of docs, two maintenance burdens.

Connect: The Inflection Point

The Connect protocol from Buf changes this equation.

Connect is simple: gRPC services that also speak HTTP/JSON natively. Same proto definition, but clients can choose their transport:

  • Internal services → Use binary gRPC for efficiency
  • Browser clients → Use HTTP/JSON, no grpc-web needed
  • External developers → Use curl, standard HTTP clients
  • AI agents → Use whatever works

One schema. Multiple transports. No translation layer.

What this means: Companies can now build with protobuf from day one and expose the same APIs externally without maintaining a separate REST interface. The "internal vs. external" distinction starts to dissolve.

AI-Assisted Protobuf Development

Here's where it gets interesting. As protobuf moves from internal-only to external-facing, the tooling needs to keep up.

Writing proto files by hand is tedious. Field numbers, syntax rules, import paths, naming conventions—there's a lot to get right. And when you're exposing APIs externally, consistency matters even more.

This is where AI-assisted development shines:

  • Natural language generation — Describe your service, get correct proto3 syntax
  • Best practices enforcement — Consistent naming, proper field numbering, backwards-compatible changes
  • Schema discovery — Find existing schemas before creating duplicates
  • Documentation generation — Keep docs in sync with your schema

Why This Matters Now

The convergence is happening:

  1. Connect makes protobuf viable for external APIs — The browser/HTTP barrier is gone
  2. AI coding tools are mainstream — Claude, Copilot, and others are how developers work now
  3. Schema-first development is winning — Teams want one source of truth, not OpenAPI + proto + docs

The question isn't whether to use protobuf for external APIs. It's how to do it efficiently.

The Opportunity

For teams already using protobuf internally:

  1. Add Connect — Expose your existing gRPC services over HTTP/JSON
  2. Use AI-assisted tooling — Generate schemas from natural language, enforce conventions automatically
  3. Publish to a registry — Share schemas across your organization for discovery and reuse

Your internal services become external APIs. Your schema definitions stay consistent. Your documentation generates from the same source.

The "internal vs. external" distinction was always artificial. Connect removes the transport barrier. AI tooling removes the productivity barrier.

What's left is just APIs—usable by humans, services, and AI agents alike.
